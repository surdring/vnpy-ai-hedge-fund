# AGENTS 通用规则

## 1) 语言
- 始终使用中文回复（除非用户明确要求使用其他语言）。

## 2) 文档生成与落盘
- 当用户要求"生成文档/规范/模板/清单"等内容时：
  - 必须根据仓库目录结构，选择**合理的目标目录**（优先 `docs/`）。
  - 必须以 **Markdown（.md）** 格式在目标目录中**创建文件并写入内容**（而不是只在聊天中输出）。
  - 若用户未指定目标目录或命名：
    - 先提出 1-3 个建议路径与文件名供用户确认，再创建文件。

## 3) 任务状态标记
- 任务**只有在自动化验证通过后**方可标记完成（避免"文档已完成但代码不可运行"）：
  - 优先运行任务说明中指定的验证命令；若未指定，则运行仓库默认验证命令（如 `npm run build`、`cargo check`、`go test` 等，根据技术栈确定）。
  - 若因环境/依赖缺失导致无法运行验证，必须在任务条目中记录原因与补验收计划，**不得直接标记完成**。
- **目标驱动执行**：将任务转化为可验证的目标：
  - "添加校验" → "先编写针对非法输入的测试，再让测试通过"
  - "修复 Bug" → "先编写复现该 Bug 的测试，再让测试通过"
  - "重构 X" → "确保重构前后测试全部通过"
- 对于多步骤任务，先陈述简要计划：
  ```
  1. [步骤] → 验证：[检查项]
  2. [步骤] → 验证：[检查项]
  3. [步骤] → 验证：[检查项]
  ```
- 强成功标准让你能独立迭代；弱标准（"让它工作"）需要不断澄清。
- 完成每个任务后，在 `docs/` 目录中生成该任务的验收日志（若用户要求）。
  - **验收日志命名**：`docs/YYYY-MM-DD_short-slug.md`
- **任务清单状态标记**：当任务完成后，须在任务清单中将对应的 `[ ]` 替换为 `[*]`。

## 4) 通用编码规范

1. **技术栈适配**
   - 严格遵循当前仓库已有的技术栈、框架版本和编码风格。
   - 不引入与现有架构冲突的依赖或范式。

2. **错误消息**
   - 错误消息（message）必须用**英文**（便于日志搜索）；注释与 UI 文案可中文。

3. **类型与模块化**
   - 类型定义应放在合适的位置，避免客户端 bundle 拉入 Node-only 模块。
   - 遵循仓库已有的目录约定和模块边界。

4. **代码安全与隐私**
   - 禁止在任何前端、日志、对话输出中泄露密钥、Token、Cookie、内部地址、账号口令。
   - 如用户提供敏感信息，必须提醒其脱敏，并在后续推理中避免复述原文。

## 5) AI 模块规范
- AI 功能**默认关闭**（不在启动时引入外部依赖）。
- 本地优先：优先支持本地推理模型（llama.cpp / Ollama 等），数据不出本机。
- BYOK 可选：支持用户自带云端 API Key，项目不内置任何 Key。
- 绝不变重：不引入数据库替代核心轻量存储（数据库仅用于日志等辅助场景）。
- Provider 接入：通过 OpenAI-compatible API 调用模型，Provider 预设配置在环境变量或配置文件中。
- API Key 脱敏：读取配置返回前端时，apiKey 仅保留前 3 位和后 4 位，中间 `****`。
- **推理策略选择原则**（参考 Chain-of-Draft 思想）：
  - **核心意图**：多步推理任务中，引导模型用极简草稿代替冗长 CoT，在保持准确率的同时显著减少 token 用量和响应延迟。
  - **Standard 策略**（简单/直接任务，如上下文摘要）：直接输出结论，不引导中间推理，避免复杂指令约束。
  - **CoD 策略**（多步推理任务，如错误诊断、复杂分析）：引导模型分步思考，但要求每一步只保留关键信息、极度精简；不硬编码具体词数限制（不同模型遵循能力差异大）。
  - **本地推理模型兼容**：部分模型会将思考过程与答案分开放置（如 `reasoning_content` / `content`），后端需兼容读取两种字段。
  - **后处理不可移除**：所有 AI 输出必须经过后处理（清理思考过程、截断、去重），兼容不同模型的输出格式差异。
  - **避免硬编码 prompt 模板**：不在代码中写死示例输出，防止模型机械模仿导致内容僵化。

## 6) 数据持久化规范
- 核心数据优先使用与项目规模匹配的轻量存储方案。
- 并发控制：读写共享资源时使用互斥锁或等价机制保护。
- 进程状态：使用端口探测为主、PID 为辅的混合模型判断进程是否存活。

## 7) 跨平台适配
- 使用策略模式封装平台差异（Linux / macOS / Windows）。
- 系统命令、目录选择等操作需适配各平台标准工具。

## 8) 文档生成失败处理
- 操作失败已被中止，在对话框中完整显示内容。
- 内容必须完整，包含所有审查维度和结论。

## 9) 行为准则（Behavioral Guidelines）

### 9.1 先思考再编码（Think Before Coding）
**不要假设。不要隐藏困惑。暴露权衡。**

实现之前：
- 明确陈述你的假设。如果不确定，提问。
- 如果存在多种解读，全部列出——不要默默选择。
- 如果存在更简单的方案，直接指出。必要时敢于提出异议。
- 如果某件事不清晰，停下来。明确指出困惑点。提问。

### 9.2 极致简洁（Simplicity First）
**用最少的代码解决问题。不做预测性编码。**

- 不实现未被要求的功能。
- 不为单次使用的代码创建抽象层。
- 不引入未被要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景编写错误处理。
- 如果写了 200 行但可以用 50 行解决，重写。
- 自问："资深工程师会觉得这个过于复杂吗？"如果是，简化。

### 9.3 精准修改（Surgical Changes）
**只改动必须改的部分。只清理自己引入的遗留问题。**

编辑现有代码时：
- 不"改进"相邻代码、注释或格式。
- 不重构没有问题的代码。
- 匹配现有风格，即使你更倾向另一种写法。
- 如果发现无关的废弃代码，可以提及——不要删除。

自己的改动产生孤儿代码时：
- 清理自己导致不再使用的导入/变量/函数。
- 不要删除预先存在的废弃代码（除非被要求）。

检验标准：每一行改动都应能直接追溯到用户的需求。
